SecondBrain assistant guide

SecondBrain is a local AI assistant for chatting with your detected local LLM, reading and managing files inside an active workspace, validating drillhole data, editing DH validation config, and running guarded read-only SQL Server queries.

Model {{ initial_state.model }}

Responses come from the detected local provider: {{ initial_state.llm_provider }}.

Workspace Workspace-bound files

File tools resolve paths inside the active workspace and reject path escapes.

Tools Read-only by default

Natural-language tools can read, search, validate, or query; file writes require slash commands.

Local LLM auto-detection

The app can use Ollama or LM Studio. By default it probes Ollama at http://localhost:11434 and LM Studio at http://localhost:1234/v1, then picks an available model.

export SECONDBRAIN_LLM_PROVIDER=auto

Force a provider

# Ollama
export SECONDBRAIN_LLM_PROVIDER=ollama
export OLLAMA_BASE_URL=http://localhost:11434
export OLLAMA_MODEL=gemma4:e2b

# LM Studio
export SECONDBRAIN_LLM_PROVIDER=lmstudio
export LM_STUDIO_BASE_URL=http://localhost:1234/v1
export LM_STUDIO_MODEL=your-loaded-model-id

Workspace and files

The workspace is the root folder the assistant is allowed to inspect for file operations. The file tree in the sidebar shows folders and files from that active workspace, excluding dependency/cache folders such as virtual environments.

Change workspace

Use the Workspace input in the sidebar, or send:

/workspace /absolute/path/to/project

After switching, chat history, file tree, memory search, and file tools use the new workspace.

Path rules

Use workspace-relative paths for file commands:

/read notes/example.md
/copy data/input.csv archive/input.csv

Paths such as ../outside.txt are rejected for workspace safety.

Natural language

You can speak or type normal requests. The agent first decides whether a safe read-only tool is useful, then summarizes the result in chat. Voice input is only used to create user messages; the assistant does not replay responses with voice.

list files in this workspace baca file README.md search memory tentang drillhole config validate lithology.xlsx ini query database tampilkan 10 customer terbaru

Natural language is intentionally conservative. It can read/list/search/validate/query, but it will not write files, copy files, move files, or export reports unless you use an explicit slash command.

Slash commands

Slash commands are explicit actions. Use them when you want deterministic behavior, file mutations, or report output.

Command Purpose Example
/workspace Switch the active workspace. /workspace /Users/me/project
/doc Create a markdown document in the workspace documents folder. /doc ringkasan validasi drillhole
/read Read a workspace file, optionally by line range. /read README.md 1 80
/write Write content to a workspace file. /write notes/todo.md Review config
/append Append content to a workspace file. /append notes/todo.md Add SQL profile
/copy Copy a file inside the active workspace. /copy data/a.csv backup/a.csv
/move Move a file inside the active workspace. /move draft.md notes/draft.md
/ls List markdown files in a workspace path. /ls docs
/search Search indexed memory content. /search userConfig
/dh Run drillhole config and validation commands. /dh validate --collar collar.csv --lithology lithology.xlsx
/sql Inspect SQL profiles, cache schema, explain, or run read-only SQL Server queries. /sql ask default show top customers

Drillhole validation skill

The DH validation skill reads CSV/XLSX files from the active workspace and validates them against the embedded secondbrain/defaults/userConfig.json. The DH Config tab edits that embedded config directly, so rules are shared across workspace switches.

Validate files

/dh validate --collar collar.csv --survey survey.csv --lithology lithology.xlsx --assay assay.xlsx

Optional tables are allowed. Cross-table rules run only when the required tables are provided.

Write a report

/dh validate --collar collar.csv --lithology lithology.xlsx --out reports/dh-report.md

Report output requires the explicit --out slash command option.

Editing userConfig.json

Open the DH Config tab. Choose a table type, edit column rules, mandatory flags, data types, numeric ranges, lookup libraries, and code descriptions. Changes stay local until you click Save. Reload discards unsaved edits.

Important fields: configs stores per-table column rules, libraries stores valid lookup codes, and lookup validation must point to an existing library.

SQL Server skill

SQL Server access is read-only and configured through environment variables. The app keeps credentials out of workspace files, UI state, LLM prompts, and chat responses.

1. Create SQL Server login

Run this once as a database administrator. Use a dedicated assistant login, not your personal/admin login.

USE master;
CREATE LOGIN secondbrain_app_user
WITH PASSWORD = 'replace-with-a-strong-password',
CHECK_POLICY = ON,
CHECK_EXPIRATION = ON;

USE YourDb;
CREATE USER secondbrain_app_user FOR LOGIN secondbrain_app_user;

2. Create read-only role

Grant only the schema or views the assistant may inspect. Avoid db_owner, db_datawriter, and broad production use of db_datareader.

USE YourDb;
CREATE ROLE secondbrain_reader;
GRANT SELECT ON SCHEMA::reporting TO secondbrain_reader;
GRANT VIEW DEFINITION ON SCHEMA::reporting TO secondbrain_reader;
ALTER ROLE secondbrain_reader ADD MEMBER secondbrain_app_user;

Prefer curated reporting views over broad roles such as db_datareader.

3. Configure app profile

SECOND_BRAIN_SQL_PROFILES is non-secret metadata. It tells the app which env var contains the real connection string and which schemas/objects are allowed.

export SECOND_BRAIN_SQL_PROFILES='{
  "default": {
    "connection_env": "SECOND_BRAIN_SQL_DEFAULT_CONNECTION",
    "schemas": ["reporting"],
    "max_rows": 100,
    "timeout_seconds": 20
  },
  "drillhole": {
    "connection_env": "SECOND_BRAIN_SQL_DH_CONNECTION",
    "objects": ["reporting.collar", "reporting.survey"],
    "max_rows": 200,
    "timeout_seconds": 20
  }
}'

4. Configure credentials

Put the actual SQL Server login/password only in the referenced connection env var. Restart the UI after changing env vars.

export SECOND_BRAIN_SQL_DEFAULT_CONNECTION='Server=tcp:localhost,1433;Database=YourDb;UID=secondbrain_app_user;PWD=your-password;Encrypt=yes;TrustServerCertificate=no'

export SECOND_BRAIN_SQL_DH_CONNECTION='Server=tcp:localhost,1433;Database=YourDb;UID=secondbrain_app_user;PWD=your-password;Encrypt=yes;TrustServerCertificate=no'

Profile fields

Field Meaning Recommendation
connection_env Name of the env var containing the real connection string. Use one secret env var per profile.
schemas Allowed SQL schemas for metadata cache and query validation. Use ["reporting"] or another curated read-only schema.
objects Optional stricter allowlist of exact schema.table or view names. Use this for sensitive databases.
max_rows Maximum result rows the app will request. Keep 100-200 for chat; app caps high values.
timeout_seconds Query timeout. Use 10-30 seconds for interactive chat.

SQL commands

/sql profiles /sql schema default --refresh /sql explain default show top 10 customers /sql ask default show top 10 customers /sql run default SELECT TOP (10) id, name FROM reporting.customers

Natural-language query examples

query database tampilkan customer terbaru lihat 20 collar terakhir dari SQL Server berapa jumlah record survey per bulan? buatkan query SQL untuk melihat assay tertinggi

The SQL guard rejects destructive keywords, multiple statements, unknown tables/columns, and queries outside the cached schema. Generated queries are limited by max_rows and run with a timeout.

First-time flow: export env vars, restart the UI, run /sql profiles, run /sql schema default --refresh, then use /sql ask, /sql run, or normal chat.

Safety model

Natural language tools

Allowed: list files, read files, search memory, validate DH files, query SQL Server read-only.

Blocked: file writes, append, copy, move, deletion, report writing, and destructive SQL.

Explicit commands

Use slash commands when an operation changes files or writes reports. Paths are still checked by workspace safety rules.

Credential handling

Use env vars for secrets. Do not put SQL passwords in workspace files, chat messages, notes, or userConfig.json.

Schema grounding

For SQL, refresh schema cache before asking questions. The agent should only query tables and columns from that cache.

Troubleshooting

Local LLM connection

For Ollama, make sure Ollama is running and a model is available:

ollama serve
ollama pull gemma4:e2b

For LM Studio, open the Local Server tab, load a model, and start the server on port 1234.

SQL profile missing

Run /sql profiles. If the profile is missing, export SECOND_BRAIN_SQL_PROFILES before starting the UI.

SQL table not found

Run /sql schema default --refresh and confirm the table appears in the cached schema whitelist.

DH validation mismatch

Check column names in the DH Config tab. Headers are normalized, but required columns still need matching rule names.
